# React Practice Project Plan

## 项目目标

这个仓库的目标，不是只做一个能运行的 React Demo，而是作为你从 Vue 技术栈迁移到 React 技术栈的第一套系统化练手项目。

这份项目计划围绕三个目标展开：

1. 用 React 完整落地一个中后台练手项目
2. 在实战中掌握 React 常用技术栈和工程结构
3. 为后续面试准备可复述的项目经验和技术总结

---

## 项目定位

建议将第一个项目定位为：

`React 中后台管理系统练手项目`

这是最适合你当前阶段的项目类型，因为它能覆盖 React 项目最常见的开发场景：

- 登录流程
- 页面布局
- 路由系统
- 列表页
- 表单页
- 状态管理
- 请求封装
- 权限控制基础
- 组件封装

相比做一个纯展示型页面，中后台项目更适合练习工程能力，也更适合面试表达。

---

## 第一版技术栈

建议第一版先不要堆太多新技术，控制复杂度，优先把主线走通。

### 核心技术栈

- `React`
- `TypeScript`
- `Vite`
- `React Router`
- `Ant Design`
- `Axios`
- `Redux Toolkit`

### 第二阶段再补充

- `TanStack Query`
- `Vitest` 或 `Jest`
- `React Testing Library`
- `Tailwind CSS`
- `Next.js`

当前阶段的原则是：

先把最核心的 React 项目能力练出来，再逐步引入更现代、更高级的生态工具。

## 目录职责说明

### `docs/`

用于记录学习过程和项目总结，不只是补充文档，而是你后续面试的素材库。

建议放：

- 学习路线
- 项目计划
- 每日记录
- 面试题总结
- 组件设计总结

### `src/api/`

用于管理接口请求逻辑。

建议拆成两层：

- `request.ts`：统一请求实例、拦截器、错误处理
- `modules/`：按业务模块拆分接口，例如 `user.ts`

### `src/components/`

放公共可复用组件，比如：

- 通用表格组件
- 搜索表单组件
- 弹窗表单组件
- 空状态组件

### `src/hooks/`

放自定义 Hook，比如：

- `usePagination`
- `useTableQuery`
- `useModal`

### `src/layouts/`

放布局组件，比如后台管理系统的主布局：

- 左侧菜单
- 顶部栏
- 内容区

### `src/pages/`

放页面级组件，按业务模块拆分。

建议第一阶段至少有：

- `login`
- `dashboard`
- `user`
- `not-found`

### `src/router/`

统一维护路由配置、懒加载和权限控制入口。

### `src/store/`

统一维护全局状态。

建议第一阶段只存这些全局状态：

- 用户登录信息
- token
- 菜单/权限基础信息
- 全局 UI 状态

### `src/styles/`

用于放全局样式、重置样式、主题变量。

### `src/types/`

用于放公共类型定义，减少类型分散和重复。

### `src/utils/`

用于放纯工具函数，例如：

- 时间格式化
- 存储封装
- 参数处理

## 分阶段执行路线

## 阶段 1：项目初始化

### 目标

把项目基础工程跑起来。

### 任务

- 初始化 `Vite + React + TypeScript`
- 安装核心依赖
- 配置基础目录
- 配置代码规范
- 配置全局样式入口

### 完成标准

- 项目可以本地启动
- 目录结构已建立
- 页面可以正常渲染

---

## 阶段 2：路由和布局

### 目标

先把中后台应用的“骨架”搭出来。

### 任务

- 接入 `React Router`
- 创建登录页
- 创建主布局页
- 创建仪表盘页
- 创建 404 页
- 实现路由跳转
- 实现菜单基础结构

### 完成标准

- 可以在不同页面间切换
- 布局结构完整
- 菜单和页面联动正常

---

## 阶段 3：用户模块

### 目标

开始进入真实业务模块开发。

### 任务

- 创建用户列表页
- 添加搜索条件区域
- 添加表格展示
- 添加分页逻辑
- 添加新增用户页面或弹窗
- 添加编辑用户功能

### 完成标准

- 用户列表页具备基本增删改查流程
- 页面结构清晰
- 表单和表格逻辑可运行

---

## 阶段 4：请求与状态管理

### 目标

补齐项目基础设施，让项目具备真实项目结构。

### 任务

- 封装 Axios 请求实例
- 增加请求拦截器
- 增加错误处理
- 配置 Redux Toolkit
- 增加用户登录状态管理

### 完成标准

- 请求逻辑统一
- 登录状态可管理
- 页面不再直接散写请求逻辑

---

## 阶段 5：优化与抽象

### 目标

开始从“能跑”提升到“更像正式项目”。

### 任务

- 抽离公共表格逻辑
- 抽离公共表单逻辑
- 封装常用 Hook
- 优化目录组织
- 补充类型定义

### 完成标准

- 重复逻辑明显减少
- 项目结构更清晰
- 页面代码可维护性提升

---

## 阶段 6：总结与面试沉淀

### 目标

把项目经验转化成可输出、可复述的能力。

### 任务

- 记录项目结构设计思路
- 总结技术选型原因
- 总结遇到的问题和解决方案
- 整理 React 面试高频问题

### 完成标准

- 能独立讲清楚这个项目
- 能把项目写进简历
- 能围绕项目回答 React 面试问题

## 推荐开发顺序

后续执行时，建议严格按这个顺序推进：

1. 初始化工程
2. 建立目录结构
3. 路由系统
4. 主布局
5. 登录页
6. 仪表盘页
7. 用户列表页
8. 新增/编辑用户
9. 请求封装
10. 状态管理
11. 公共组件抽象
12. 文档总结

这样安排的原因是：

- 先让应用骨架跑起来，建立成就感
- 再做业务模块，积累 React 页面开发经验
- 最后补抽象和优化，避免一开始设计过重
